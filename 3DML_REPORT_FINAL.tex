\subsection{Early stopping}
We made an addition to the code, we monitor validation/test loss every epoch and keep the best checkpoint (lowest test loss). Training stops when no improvement is observed for a 30 epochs. This helps prevent over-training. The maximum number of epochs in our experiments was 120 epochs. We tried 250 epochs but it gave almost no benefit.

\section{Exercise 2.3 : Data augmentation for 3D data}
\subsection*{Question 1: Proposed augmentation}
We propose \textbf{RandomPointDropout}:
for each training point cloud, a random subset of points is replaced/dropped.
This simulates partial observations (occlusion/sparse scans) and reduces over-reliance on a few dominant points selected by max pooling.

\section{Best result model (data augmentations and architectural changes)}
\label{final_run}
\subsection{Architecture}
During this final run, we combine the learnings from the additional esperiments section ~\ref{additional_runs}, we use the input only T net PointNet as a baseline.
The classifier head uses \textbf{double dropout}:
\begin{itemize}[leftmargin=*]
  \item \texttt{Dropout(0.3)} after \texttt{fc1} + BN + ReLU,
  \item \texttt{Dropout(0.3)} after \texttt{fc2} + BN + ReLU.
\end{itemize}

This made the training more stable.

\subsection{Data augmentation}

Compared to the FullPointNet we used previously which uses a classifier dropout of 0.3, random z-rotation, point shuffle and gaussian noise, this final architecture adds an other dropout (one after fc1 and the other after FC2). For data augmentation, we add unit sphere normalization (for each point cloud, we center it at the origin and then rescale it so that the furthest point is at a distance of 1), changing z-rotation to be probabilistic with a probability of 0.3 and a smaller noise ($\sigma$ being 0.005) and a smaller clip of $\pm0.02$. Finally, we add RandomScaleShift 
\[
\tilde{\mathbf{x}}_i = s\,R_z(\theta)\,\mathbf{x}_i + \mathbf{t} + \boldsymbol{\epsilon}_i.
\]

with $s\in[0.9,1.1]$ which is the random scale factor and $t\in[-0.03,0.03]^3$ which is the random translation vector and a probability of 0.3.



For this model, we took inspiration from PointNet articles:  \cite{pointnet,revisit_pointcloud}.


\textbf{Note:} we use a fixed seed : seed=101, Adam (lr $=10^{-3}$, weight decay $=5\times10^{-5}$), cosine LR scheduler (min lr $=10^{-5}$), batch size 32, workers 8 and a train test split of 80-20\%


With these settings, we reach the best performance in the test set at epoch 90 with 90\% of accuracy.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.95\linewidth]{pointnetfull_training_curves_2.png}
  \caption{Training and validation loss for the last run, 90\% accuracy}
\label{fig:last_best_run}
\end{figure}

\subsection{Discussion: }
Using this model gave the best results. This is because we combined different additions from normalization and small geometric perturbations which helped robustness without destroying class geometry, double dropout that regularized the classifier head better than only one dropout and avoided overfitting. Scale shift improved generalization where features won't be dominated by coordinate magnitude differences.



\section{Additional experiments}
\label{additional_runs}
\subsection{Optimization and augmentation study outside architecture changes}
We added multiple runs while changing one thing at a time to find which architectural change gives the best results. 

\begin{figure}[H]
  \centering
  \includegraphics[width=0.95\linewidth]{selected_runs_acc_nll.png}
  \caption{what helped and what didn't in the runs}
\label{fig:selected_trajectory}
\end{figure}
\subsection{Discussion of the runs (Figure~\ref{fig:selected_trajectory})}

\begin{enumerate}[leftmargin=*]
  \item \textbf{adding normalization + random sample:} Adding normalization and random sample reduced the accuracy from 86.10 to 85.41. This change isn't useful.
  \item \textbf{r01 $\rightarrow$ r02 (add scale/translate/jitter/point-dropout):} Adding scale which multiplies the XYZ coordinates by a random scalar, translate which adds a random shift vector with each axis (in order for the object to not be always centered in the same way), random jitter adds per point gaussian noise to reduce overfitting to the coordinates, finally, point dropout drops some points to push the model to learn global shape instead of some key points. This gave a +1.34 gain in accuracy which means that geometric perturbations add better generalization.
  \item \textbf{r02 $\rightarrow$ r03 (switch to AdamW + cosine + smoothing + clipping):} The accuracy didn't see a big gain with these changes (only +0.28).
  \item \textbf{disable train-time z-rotation:} accuracy improved to 88.53. This supports the hypothesis that for aligned ModelNet40, extra rotation augmentation may conflict with class-discriminative orientation cues.
  With this change, the accuracy improved to over 88\% which means that more rotation augmentation can hurt when the discremenating factor between classes is orientation.


\end{enumerate}

Conclusion: Changing only one thing doesn't add a lot of gains. But when combining multiple gains, we can see a substancial gain as we'll see in the final architecture.

\textbf{Note: } The code for these runs is in ex2\_full\_optimization\_runs.py 

\subsection{PointNet with feature T-Net $64\times64$}
Using the combined gains with the run r16 trained with AdamW (lr=1e-3, weight\_decay=1e-4), cosine LR scheduling (eta\_min=1e-5), label smoothing (0.1), gradient clipping (1.0), and augmentation set to rotate\_z=False scale\_translate=True, jitter=True, point\_dropout=False (with normalize=False and random\_sample=False), and it ran up to 120 epochs with the best checkpoint reached 89.55\% test accuracy. This is then used to compare between having the input only T net (3*3) and the Tnet with 64*64+3*3 where we arrived at the same result (89.51\%): 

\begin{itemize}[leftmargin=*]
  \item (3x3 + 64x64): 89.51\%, NLL 0.4421
  \item (3x3 only): 89.55\%, NLL 0.4384
\end{itemize}
Difference is very small in this comparison.

\section{Conclusion}
At the end of the project, after combining the different improvements, we achieved a score of 90\% on the 40 class modelnet dataset. We found that a combination of data augmentations (scale shift, double dropout and normalization to the dataset). From the runs, we found that the full pointNet architecture with an additional 64x64 T net didn't really help with the performance. PointNet yielded a much better performance than PointMLP mainly due to the fact that PointNET explains the shape of the data better than a MLP that flattens it. Finally, while 90\% accuracy is an acceptable result in the dataset, more modifications (or a new architecture) need to be implemented to reach "production" ready performance where the model can be better trusted.
